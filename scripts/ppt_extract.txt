--- Slide 3 ---
问题陈述
为什么传统的智能体评估是不充分的：
仅离线评分
大多数评估框架都是针对静态数据集或模拟环境对智能体进行打分。它们无法反映智能体在生产环境中面对的真实浏览器状态化 UI 交互。
可复现性差
如果没有受控的 RPA 执行层，运行结果就会不具确定性。轨迹回放无法实现，使得调试和回归测试都不可靠。
可观测性弱
现有工具缺乏结构化的轨迹日志、步骤级指标以及恢复事件捕获。故障表现不透明，难以系统化诊断。
缺乏目标驱动的动态流程
静态测试脚本无法在运行时根据目标意图自适应。传统框架缺少动态办公/智能体发现和基于主题的选择机制。
Offline Scoring
Poor Reproducibility
Low Observability
No Goal-Driven Flow
Missing Recovery Metrics

--- Slide 4 ---
RAFT 是什么：系统角色模型
RAFT 定义了三个主体之间清晰的角色分工：
RAFT = 测试者
RAFT 负责编排测试场景。它构建目标、选择代理、安排运行，并推动评估生命周期。它是考官——而不是被测试的代理。
目标代理 = 被考察者
目标代理（基于 LLM 或规则）接收目标查询，并必须在 Poffices UI 中导航以完成任务。它的决策、步骤和结果都会被观察和评分。
RPA = 环境执行器
Playwright RPA 在真实的 Poffices 浏览器环境中执行代理的动作。它在每个动作后提供真实状态反馈，闭合运行中的反馈循环。
评估 = 评审者
评估层使用核心指标（成功、步骤数、恢复次数、超时次数）以及可选的 LLM 评审维度对每次运行进行评分。它会为每条轨迹生成最终判定。

--- Slide 8 ---
四层架构
第 4 层：评估层
对轨迹进行评分。计算成功率、步数、超时次数、恢复次数。可选的 LLM 裁判。为跨运行循环提供反馈。
第 3 层：智能体与 RPA 层
目标智能体决策 + Playwright RPA 执行。闭合运行内循环：决策 → 动作 → 状态更新 → 下一次决策。
第 2 层：编排器层
按智能体调度运行，组合查询上下文（前几轮、策略提示），在失败时管理重规划，协调多智能体流程。
第 1 层：实验 / 任务层
加载配置和任务定义。解析目标意图。驱动动态办公环境 / 智能体发现。入口：
run_poffices_agent.py
。

--- Slide 12 ---
统一执行入口：
run_poffices_agent.py
所有 RAFT 执行流程都通过一个统一的入口脚本驱动，为所有模式和配置提供一致的 CLI 接口：
--runs
每个 agent 执行的运行次数。支持多次运行的统计汇总。
--goal
自然语言目标字符串。由 B1（目标解释器）解析，以提取办公意图、主题和 agent 数量。
--agents
显式 agent 列表覆盖。未指定时，将通过主题匹配进行动态发现。
--strategy
传递给编排器的查询策略提示，用于跨运行上下文构建。
--full-report
触发生成完整 HTML 报告（
run_report.html
），包含所有轨迹工件和汇总层。
CLI 能力映射
示例调用：
# 单个 agent，3 次运行，完整报告
python run_poffices_agent.py \
 --goal "Test HR office scheduling topic" \
 --runs 3 \
 --full-report

# 多个 agent，动态发现
python run_poffices_agent.py \
 --goal "Evaluate 2 agents on finance topic" \
 --runs 5 \
 --strategy adaptive \
 --full-report

# 显式 agent 覆盖
python run_poffices_agent.py \
 --agents agent_a agent_b \
 --runs 2


--- Slide 13 ---
执行模式
RAFT 支持一个执行配置矩阵，以覆盖多样化的测试场景：
单智能体 / 单次运行
基线模式。一个目标智能体，一次运行。用于快速迭代和调试。产生一条轨迹日志和一条指标记录。
单智能体 / 多次运行
统计模式。一个目标智能体，N 次运行。聚合各次运行的成功率、平均步数、超时率、恢复率。支持趋势检测。
多智能体 / 单次运行
对比模式。多个目标智能体，每个运行一次。并排进行指标比较。适用于 A/B 智能体评估。
多智能体 / 多次运行
完整评估模式。多个智能体 × N 次运行。提供包含每个智能体和汇总结果的完整 HTML 报告。是质量门禁的主要模式。

故障处理与恢复
恢复序列
01
检测到故障
RPA 操作失败或 
get_response
 超时。故障事件会连同时间戳和错误类型一起记录到轨迹中。
02
触发重规划
协调器检查 
replan_count < max_replans
。如果在限制范围内，则会构造并注入恢复计划。
03
执行恢复操作
标准恢复序列：
refresh_page
 → 
wait_output_complete
 → 
get_response
。每一步之后都会重新评估状态。
04
恢复或中止
如果恢复成功：运行从已恢复状态继续。如果超过最大重规划次数：运行将被标记为失败，并保留完整诊断日志。
为什么这能提升鲁棒性
故障绝不会是静默的——每个故障事件都会连同上下文一起记录
恢复尝试会被计数（
recovery_count
 指标）以便观测
最大重规划限制可防止无限循环
恢复序列是标准化且可重复的
诊断日志支持按故障类型进行根因分析

失败场景
故障点
get_response
 在第 [N] 步运行 2 时超时。Agent 已向 Poffices UI 提交查询，但响应未能在配置的超时时间窗口（[T seconds]）内返回。
触发的恢复操作
启动重新规划 #1：执行 
refresh_page
 → 调用 
wait_output_complete
 → 重试 
get_response
。刷新后重新评估状态。
最终结果
在重新规划 #1 后恢复成功。运行从恢复状态继续。任务成功完成。总恢复开销：[T seconds]。在指标中将该运行标记为“恢复成功”。
诊断
根本原因：Poffices UI 在高负载下响应延迟激增。缓解措施：在高延迟环境中提高

权衡：真实性 vs 速度
LLM 调用
最大的时间驱动因素。LLM 辅助模式会增加延迟，但能实现目标驱动的动态流程。基于规则的模式更快，但适应性较弱。
wait_output_complete
对于真实环境保真度是必要的。超时阈值是关键调优参数——过低会增加误判失败，过高则会拖慢吞吐量。
重试 / 恢复
恢复会为每次事件增加 [T seconds] 的额外开销，但可以防止静默失败。对质量的净影响是正面的；对速度的净影响是负面的。
